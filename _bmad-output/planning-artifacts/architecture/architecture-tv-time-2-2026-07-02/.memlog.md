- (constraint) [ADOPTED PRD NFR2] Client is Expo/React Native, single codebase, iOS + Android v1 feature parity. Web explicitly post-v1. F-Droid Android is a build variant of the same codebase (Google-free push/auth path only, not divergent product behavior).
- (constraint) [ADOPTED PRD NFR3] F-Droid build ships zero proprietary Google/Firebase/Play Services deps -- constrains auth, push, analytics for that variant only. Play Store Android + iOS may use FCM/APNs.
- (constraint) [ADOPTED PRD Goals] Single centrally-hosted multi-user instance; self-hosting/federation explicitly out of v1 (post-v1 survivability path only).
- (constraint) [ADOPTED PRD Overview] No revenue, ever -- solo maintainer, zero budget. Hosting/infra must be near-zero-cost / free-tier-friendly and low-ops.
- (constraint) [ADOPTED PRD NFR6] EU-based creator + users -> GDPR applies: lawful basis, export (FR4), deletion (FR5), breach handling.
- (decision) [ADOPTED addendum] Watch is the atomic timestamped unit; rating/mood/note attach per-watch, never as a single mutable field on a title -- repeat watches of the same title each keep independent reactions (FR20).
- (decision) [ADOPTED addendum H2, NFR9] Catalog traffic (TMDB or equivalent) is proxied through the backend -- API key never ships in the open-source F-Droid client binary.
- (decision) [ADOPTED FR37, addendum] Push transport is per-build-variant: APNs (iOS), FCM (Play Store Android), UnifiedPush (F-Droid Android), with graceful in-app what's-new degradation when no push channel is available.
- (decision) [ADOPTED FR39/FR40] Import is idempotent and non-destructive, keyed on a stable (title, episode, source-id/timestamp) tuple; never fabricates missing timestamps -- flags undated/approximate instead.

- (decision) Backend runs on self-hosted Supabase (official docker-compose distribution: Postgres + GoTrue auth + PostgREST + Storage + Edge Functions + Kong gateway), not hosted Supabase SaaS and not a hand-rolled service. Resolves OQ#3 (auth): GoTrue email/password + magic-link, Google-free by construction -- F-Droid-safe on all build variants.
- (direction) User confirmed: wants Supabase's DX/velocity but dockerized/self-hosted, not the hosted SaaS -- prioritizes forkability/no-vendor-billing-dependency over convenience of managed Supabase.
- (decision) Resolves OQ#4 (hosting): the self-hosted Supabase docker-compose stack runs on a generic self-managed/custom VPS (provider not brand-locked in the spine -- any Docker-capable Linux VPS). Backend + Postgres co-located on that VPS, not a managed cloud DB.
- (decision) Self-hosted Supabase stack trimmed to: Postgres + GoTrue (Auth) + Storage + Edge Functions + Kong gateway + PostgREST. Realtime explicitly excluded -- no feature requires websockets/live push (feed refreshes on open, no chat/presence). Reduces the ops surface the solo maintainer runs on the VPS.
- (decision) Paradigm: BaaS-core with RLS-as-authorization. Postgres is the single source of truth AND the authorization boundary (Row-Level Security implements FR29/FR29a private-by-default visibility, never an app-code check). PostgREST auto-generates CRUD over RLS-protected tables; GoTrue owns identity; Edge Functions are the ONLY home for logic that can't be RLS-guarded CRUD (catalog proxy/cache, TV Time import, notification poller/fan-out, export/delete). Client is feature-modular RN (one module per nav tab: home/diary/add/feed/profile), talks only through supabase-js CRUD or typed Edge Function calls -- never raw fetch to third parties from the client.
- (decision) Offline/local durability (FR14, NFR8): client keeps a local durable outbox (expo-sqlite table pending_watches) for watch-commit mutations -- UI writes there first (instant, network-independent), a sync worker drains it to Postgres via PostgREST on reconnect, optimistic reconciliation. Read-side server-state caching via TanStack Query with persisted cache for instant cold-open Up Next (NFR1). Explicitly NOT a full offline-first sync framework (WatermelonDB) -- NFR8 caps v1 offline at 'basic, not optimized'.
- (decision) Visibility/privacy (FR29/FR29a) enforced via Postgres RLS policies, not app code: every owner-scoped table carries owner_id + nullable per-row visibility override; RLS SELECT policy = owner OR (requester follows owner AND effective visibility [row override ?? owner's global share_activity toggle] = shared). PostgREST exposes tables directly so RLS is the real boundary.
- (decision) Notification fan-out (FR33-37, addendum H3): pg_cron (bundled in self-hosted Supabase Postgres image) invokes a poll-new-episodes Edge Function daily via pg_net HTTP call over Docker-internal networking (not localhost -- self-hosted networking gotcha). The function walks distinct tmdb_show_ids with an active notify_bell, diffs against last-seen episode air-dates via the catalog cache, and fans out to each enabled push_devices row via the platform-appropriate channel (APNs/FCM/UnifiedPush) recorded on that device. Bounded daily cadence, not real-time, per addendum's own steer.
- (decision) Catalog access (FR6-9, NFR9, addendum H2): Edge Functions catalog-search / catalog-title proxy TMDB, hold the API key server-side only. Results cached in a Postgres catalog_cache table (tmdb_id, payload jsonb, fetched_at) with a TTL so FR6 stays fast under load without hammering TMDB on every keystroke.
- (decision) TV Time import (FR38-41) is an Edge Function (import-tvtime) that parses the uploaded export server-side and idempotently upserts into watches via a unique constraint on (user_id, mapped-title/episode id, source_watch_id) -- satisfies FR40's hard idempotency requirement at the DB level, not app-logic dedup.
- (decision) GDPR export/delete (FR4/FR5, NFR6) are dedicated Edge Functions (export-my-data, delete-my-account) -- export dumps all owner-scoped rows to JSON, delete cascades via ON DELETE CASCADE FKs to auth.users.id. Not ad hoc per-table app-level loops -- keeps the deletion guarantee structurally enforced.
- (constraint) Operational envelope: nightly automated off-VPS Postgres backup (pg_dump or volume snapshot shipped off-box) is a hard requirement, not optional -- the NFR11 wind-down pledge and 'no history lost' promise structurally depend on backups existing, not just on the VPS staying up. Single VPS = single point of failure for compute; backups are the mitigation until a post-v1 survivability path exists.
