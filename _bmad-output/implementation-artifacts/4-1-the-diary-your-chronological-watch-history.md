# Story 4.1: The Diary — your chronological watch history

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want a Diary of everything I've watched with my ratings, moods, notes, and dates,
so that my history reads like an autobiography.

## Acceptance Criteria

1. **Given** my watches, **when** I open Diary, **then** they show in reverse-chronological order with title/episode, star rating, mood chips, note, and date (FR22). [Source: epics.md#Story-4.1]
2. **Given** the multi-watch model, **when** a title was watched more than once, **then** each watch appears as its own dated entry with its own reaction — none collapsed into a single score (FR20, ARCH-3). [Source: epics.md#Story-4.1]
3. **Given** RLS, **when** Diary loads, **then** it reads only my own rows through the PostgREST owner policy (ARCH-5). [Source: epics.md#Story-4.1]
4. **Given** an empty Diary, **when** I have no watches, **then** the warm empty state shows ("Nothing logged yet — tonight's episode is your first entry.") linking to (+) (FR24, UX-DR14). [Source: epics.md#Story-4.1]
5. **Given** a Diary entry, **when** I tap it, **then** I can edit or remove it (reusing Story 3.7). [Source: epics.md#Story-4.1]

## Dev Notes

### Why this is a data-layer *extension*, not a new module

`app/data/watchEdit.ts` already owns "read/edit/remove a logged watch." `getWatchesForTitle` and this story's `getDiaryPage` are the same underlying read (a `watches` select, decoded the same way) with a different filter — one title vs. all of the caller's own titles. Story 3.7's own Open Questions section flagged this exact generalization as expected future work: *"worth confirming `getWatchesForTitle`'s per-title scoping is easily generalizable to a per-user, cross-title Diary query (likely just dropping the `tmdb_id`/`media_type` filters)."* That prediction holds, with one addition this story's Task 1 makes explicit: pagination. `getWatchesForTitle` could get away with a flat `.limit(200)` because it scopes to one title (bounded by how many times one person rewatches one show). Diary has no such bound — it is the user's *entire* history — so a flat limit would silently truncate a long-time user's oldest watches. Hence the keyset cursor.

### Why a keyset cursor, not `.range()` offset pagination

This codebase has no prior pagination precedent to follow blindly — `RecommendationsScreen.tsx`'s pager uses a page-number query param against a *stateless, server-computed* TMDB `/discover` feed (`catalog-recommendations`'s own pages don't shift under insert/delete the way a live table does). Diary paginates a live, mutating table: a bulk-log (3.4) or a new organic watch can insert rows *while* a user is scrolled partway through their Diary, and an edit/remove (3.7, reused here) can also fire mid-session. An offset-based `.range(20, 39)` would shift under any of those — a newly-inserted row above the current scroll position pushes every subsequent offset down by one, producing a skipped or duplicated row at the next page boundary. A keyset cursor (`watched_at`/`id` compound, matching the query's own `ORDER BY`) is immune to inserts/deletes above the cursor position by construction — the next page is always "everything after this row," not "everything at this numeric offset." This is a correctness requirement given AC1/AC2's combination (chronological + multi-watch, i.e., genuinely unbounded row growth over time), not a premature optimization.

### Why enrichment must dedupe by title, not by row

Every prior shelf (2.4's Watchlist, 3.1's Up Next, 3.8's Recommendations) already accepted "one `fetchTitleDetail` call per shelf item" as a known, explicitly-deferred fan-out concern (see `_bmad-output/implementation-artifacts/deferred-work.md`'s 2.4/3.1 entries) — acceptable there because each shelf's *items are already distinct titles* (a watchlist has no duplicate entries; Up Next tracks each show once). Diary's rows are **not** distinct titles by construction (AC2 forces every rewatch and every bulk-logged episode to be its own row) — an un-deduped enrichment pass would call the exact same `catalog-title` invoke redundantly, multiple times in the same page load, for the exact same cached response. This is a correctness/waste issue specific to this story's data shape, not the same "revisit if it ever needs to scale" tradeoff those shelves accepted — Task 2 makes the per-title dedupe a required part of the implementation, not optional polish.
