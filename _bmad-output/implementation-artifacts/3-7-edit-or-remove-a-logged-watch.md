# Story 3.7: Edit or remove a logged watch

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want to correct or delete a logged watch,
so that my history stays trustworthy when I mistap.

## Acceptance Criteria

1. **Given** a logged watch in the Diary or title detail, **when** I edit its date, rating, moods, or note, **then** the change persists to that `watches` row via PostgREST, guarded by owner-only RLS (FR16, ARCH-5). [Source: epics.md#Story-3.7]
2. **Given** a logged watch on a tracked show, **when** I remove it (including un-logging an episode *behind* the current pointer), **then** the watch is deleted and the pointer is recomputed from the full remaining watch set via the single-writer pointer RPC — which derives, never increments, so it can move the pointer backward correctly and is never client-computed (FR16, ARCH-9). [Source: epics.md#Story-3.7]
3. **Given** any edit/remove, **when** performed, **then** it is reachable tap-to-act with no long-press required (FR44). [Source: epics.md#Story-3.7]

## Scope wall — read before writing any code

This story owns **the data-layer edit/remove capability** and **one reachable entry point to it**. It does not add everything the epic's AC1 text could imply.

- **No Diary UI.** AC1's epics.md text says "in the Diary or title detail" — `DiaryScreen.tsx` is still the Story-1.3 themed placeholder (`<PlaceholderScreen title="Diary" .../>`, Epic 4/Story 4.1's job, still `backlog`). There is no chronological watch list to edit from there. **Title detail is the sanctioned entry point** for this story — the AC text itself offers it as the alternative, and unlike 3.6 (which had an existing rating-prompt surface to extend), this story has to build its entry point from scratch because nothing in the app today lists a title's individual watches anywhere. Do not attempt to stub or scaffold `DiaryScreen.tsx` — that screen's real build is Epic 4's, and epics.md's own Story 4.1 text (`**Then** I can edit or remove it (reusing Story 3.7)`) already plans to reuse this story's data layer from there, not duplicate it.
- **No new "already watched" badge on the episode list.** 3.4's scope wall already declined this for `SeasonRow`'s episode rows; this story's new "Your watches" section (Task 3) is a *separate* list below the seasons section, not a change to how episodes render.
- **No rewatch-merging or watch-count display.** AD-3 (rewatch is legitimate — each watch is its own row) is unchanged; this story lists each watch as its own row, full stop. No "watched 3×" rollup, no dedup UI.
- **No offline queue for edits/removes.** AC1's own text says "via PostgREST" (not "via the outbox") — unlike the original commit (AD-4, `logWatch`/`setWatchReaction`), an edit or remove of an *already-logged* watch is a direct network write, same posture as `watchlist.ts`/`trackedShows.ts` (no `pending_watches` row, no `getDb()`, no `triggerSync()`). If the network is down, the edit/remove fails and the UI shows a save-failed message — there is no local-first fallback here, and none is asked for by any AC (contrast FR14, which *is* explicitly network-independent and belongs only to the original log).
- **No editing of `tmdb_id`/`media_type`/`tmdb_episode_id`.** AC1 names exactly four editable fields: date, rating, moods, note. Which title/episode a watch is *for* is immutable — editing that would be "delete and re-log," not "edit," and no AC asks for it. This also means an edit never needs a pointer recompute (see Dev Notes) — only remove does.
- **No bulk remove.** Each watch is removed individually, one tap-to-act per row (AC3). `BulkLogSheet.tsx` is unrelated and untouched.
- **No new DB migration.** `watches.watched_at`/`rating`/`mood`/`note` already exist and are already owner-scoped by RLS (`0003_watches.sql`); the existing `watches_update_own`/`watches_delete_own` policies already permit exactly this story's writes. Nothing to add.
- **A test framework.** Restated every story since 1.3. Verification is a `smoke-check.mjs` extension, same as every prior Epic 3 story.

## Dev Notes

### Why edit never touches the pointer, but remove always might

AD-10's pointer is derived purely from *which episodes have a `watches` row* (`recompute_next_episode_pointer`'s Step 5/6: the set of `tmdb_episode_id`s in `watches`, walked against the chronological episode order). Editing a watch's date/rating/moods/note changes none of that set — the episode is still logged, just differently annotated — so Task 1b's `editWatch` correctly never calls the RPC. Removing a watch changes the set (one fewer watched episode), which is exactly the case AD-10/AC2 calls out: the pointer must move backward to that now-unwatched episode if it's chronologically earlier than wherever the pointer currently sits — something no prior story's code path has ever triggered (3.1 only ever advances forward from nothing; 3.2/3.4 only ever advance forward from a watch). This story is the first to exercise the RPC's "derive, don't increment" contract in the direction that actually requires deriving rather than just incrementing — hence Task 5's dedicated backward-pointer smoke check.

### Why this story is direct-PostgREST, not outbox (AD-4 does NOT apply here)

AD-4's binding language covers "the log flow (FR14) [and] the rating/mood/note prompt that immediately follows it (FR17-21)" — the *original* commit and its *immediately following* reaction, both of which need to survive a network drop because they're the sub-15-second core loop (NFR8, FR14). Editing or removing an *already-logged* watch, sometime later, from title detail, is a different user action with no such AC — AC1's own text says "via PostgREST" (matching `watchlist.ts`/`trackedShows.ts`'s established direct-write module shape), not "via the outbox." Building this on `pending_watches`/`triggerSync()` would be over-engineering a capability no AC asks for and would also raise real correctness questions this story doesn't need to solve (e.g., what does "offline-edit a watch that's also mid-sync" even mean) — see Scope wall.
