## Deferred from: code review of story-1.1 (2026-07-03)

- **Edge Functions may never report healthy → fails the AC1 "six healthy" gate.** Two intertwined causes: (a) the `functions` healthcheck uses `bash` + `/dev/tcp` which the Deno-based `edge-runtime` image may not contain [`supabase/docker-compose.yml` ~L1838]; (b) `main/index.ts` imports `jose` from a live `deno.land/x` URL with no lock/vendor, so a cold/offline first bring-up leaves `main` unable to load [`supabase/functions/main/index.ts` L1929]. Both keep `functions` unhealthy, failing AC1 and `pnpm run verify`. Options if revisited: vendor/pin `jose` + make the healthcheck `sh`-compatible; document a network prereq for first boot; or drop `functions` from the smoke-check `EXPECTED` set. — **Deferred:** app home screen already gets a Supabase 200, so the stack connects fine in practice; the functions-healthy gate isn't an active blocker.

## Deferred from: code review of story-1.4 (2026-07-04)

- **`catalog_cache` has no eviction — unbounded growth.** The TTL (`CACHE_TTL_MS`) only decides whether a row is re-written; nothing ever DELETEs stale rows [`supabase/migrations/0002_catalog_cache.sql`; `supabase/functions/catalog-search/index.ts` L192-226]. — **Deferred:** the cache is by-design "freely evictable"; a TTL sweep (pg_cron) is a separate maintenance concern, out of 1.4 scope. - **Both TMDB keys empty is masked as a connectivity 502.** With neither `TMDB_ACCESS_TOKEN` nor `TMDB_API_KEY` set, the v3 path sends `api_key=''`, TMDB 401s, and the caller sees the generic "check your connection" copy — an operator misconfiguration is indistinguishable from a user outage [`supabase/functions/catalog-search/index.ts` L64-90, L670]. — **Deferred:** hardening only; a real deploy sets the key. Non-blocking.

## Deferred from: code review of story-1.6 (2026-07-05)

- **`psql -t -A` output could miscount a stray NOTICE/banner line as a violating table.** Both new audit queries in check 8 split every non-empty stdout line and treat it as a table name; an interleaved NOTICE/warning on stdout (rather than stderr) would be misread as a real violation [`scripts/smoke-check.mjs:274`]. — **Deferred:** low-probability with the current client/server settings; not exploitable, just a false-failure risk.
- **Sign-in transient errors (429/500) are conflated with "account doesn't exist yet."** `getOrCreateSession` falls through to signup on any non-ok sign-in response, not just invalid-credentials; a rate-limit or transient 5xx on an existing account surfaces as a misleading "sign-in and sign-up both failed" error, masking an auth-service outage [`scripts/smoke-check.mjs:356`]. — **Deferred:** local dev script; low likelihood, and the true cause is still visible in the underlying HTTP status if investigated.
- **Pre-existing smoke accounts with a stale/different password produce an opaque, non-actionable failure.** If `smoke-test-a@tv-time-2.invalid`/`smoke-test-b@tv-time-2.invalid` were ever created with a password other than the current `SMOKE_PASSWORD` constant, sign-in and signup both fail with a generic error that doesn't point at "reset/delete this specific account" [`scripts/smoke-check.mjs:342`]. — **Deferred:** same root cause family as the transient-error item above; would only occur after a manual password change to a fixed synthetic account.
- **Guarded DO block assumes `public.watches` already exists.** The `pg_constraint` existence check in `0004_visibility.sql` only guards against re-adding the constraint on rerun, not against `watches` never having been created; a standalone `psql -f 0004_visibility.sql` run before `0003_watches.sql` throws a hard "relation does not exist" error [`supabase/migrations/0004_visibility.sql:32`]. — **Deferred:** matches the existing convention in `0001`–`0003` (none guard against out-of-order standalone application); the whole-folder-reapply workflow (`pnpm run supabase:migrate`) makes this moot in practice.
- **No protection against concurrent smoke-check runs sharing the same fixed smoke-test identity/row.** Two concurrent invocations against the same shared instance both target the identical fixed watch-row UUID and fixed user A/B accounts; safe today only because nothing deletes the row [`scripts/smoke-check.mjs:382`]. — **Deferred:** accepted design tradeoff (idempotency via reuse, not cleanup, per the story's own spec); revisit if this check is ever run against a shared non-local environment.
- **Hardcoded smoke-test credentials/UUIDs committed to the repo, with no guard restricting the check to a local-only `baseUrl`.** `SMOKE_PASSWORD` and the fixed emails/UUID are plain constants in `scripts/smoke-check.mjs`; nothing stops the script from creating real, permanent, known-password accounts if pointed at a non-local `baseUrl` [`scripts/smoke-check.mjs:333`]. — **Deferred:** matches the story's explicit design (fixed accounts for idempotent reruns); worth a local-only guard if this script is ever wired into a shared/staging environment.

## Deferred from: code review of story-2.4 (2026-07-06)

- **Unbounded parallel fan-out on the Watchlist shelf.** `HomeScreen.load()` runs `rows.map((row) => fetchTitleDetail(...))` through `Promise.allSettled` with no concurrency cap, batching, or pagination — an N-item watchlist fires N simultaneous `catalog-title` Edge Function invokes (each bounded at 20s), and this re-runs on *every* tab focus. A large watchlist (50+) risks rate-limiting / thundering-herd on the shared proxy and a slow, mostly-nothing shelf [`app/features/home/HomeScreen.tsx:145-147`]. — **Deferred:** the story's own Dev Notes explicitly punt this ("flag it as a future optimization candidate … if the shelf ever needs to scale past a handful of items, do not build it now"); the correct fix (a batch/lightweight catalog endpoint or a concurrency limiter) is an architecture-level decision out of 2.4's scope.
