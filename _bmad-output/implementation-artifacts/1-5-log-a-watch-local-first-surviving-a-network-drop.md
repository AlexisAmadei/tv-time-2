# Story 1.5: Log a watch, local-first, surviving a network drop

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a user,
I want to log that I watched something and have it stick even with no connection,
so that logging is truly faster than forgetting and never lost.

## Acceptance Criteria

1. **Given** a title from search, **when** I log a watch, **then** a row is written synchronously to a local `expo-sqlite` `pending_watches` table before any network call. [Source: epics.md#Story-1.5; AD-4]
2. **Given** the network is disabled, **when** I log a watch, **then** the commit still succeeds locally with a soft confirmation ("Logged — nice one."). [Source: epics.md#Story-1.5; FR14, EXPERIENCE.md#State Patterns]
3. **Given** connectivity returns, **when** the sync worker runs, **then** it drains the outbox into a `watches` row via PostgREST (uuid PK, `user_id` FK, `tmdb_id`, nullable `tmdb_episode_id`, `watched_at timestamptz`). [Source: epics.md#Story-1.5; FR15, AD-4, ARCH-10]
4. **Given** a created watch, **when** persisted, **then** rating/mood/note columns exist on the row but are null (populated in Epic 3), and the row is not modeled on a titles/catalog entity. [Source: epics.md#Story-1.5; AD-3]
5. **Given** the `watches` table, **when** created, **then** it ships with an owner-only RLS policy (`user_id = auth.uid()`) and a nullable `visibility` column defaulting to null. [Source: epics.md#Story-1.5; AD-1]

Additional behavior that must hold for the feature to work end-to-end in the existing system (not new scope, but required for the ACs to be real):

6. **Given** a search result tapped in the Add tab, **when** the tap fires, **then** it commits a watch immediately (`watched_at = now()`) — this is the walking-skeleton log action that lifts 1.4's "results are inert" scope wall. No episode/season picker, no rating/mood/note prompt, no tracked-show/next-episode-pointer wiring — those are Epic 2/3. A film and a TV show both log the same way in 1.5: `tmdb_episode_id` stays null for both (episode-level granularity is Epic 3, Stories 3.1/3.2). [Source: epics.md#Epic-1 summary "log a watch that survives a network drop"; PRD FR12 scope boundary vs FR10/FR11]
7. **Given** a watch that already synced (its `watches` row exists), **when** the sync worker retries after a partial failure (e.g. the insert succeeded but the response was lost to a network drop), **then** the retry does not create a duplicate or error the whole batch — sync is idempotent, keyed by a client-generated `id`. [Source: ARCHITECTURE-SPINE.md#AD-4 "the client never sends a mutation... without first confirming which state the row is in"]
8. **Given** the `pending_watches` local table, **when** the app is opened for the first time on a device, **then** it is created via an idempotent local migration (mirrors the server-migration discipline — safe to re-run, never assumes a fresh install). [Source: derived from AD-4 + existing migration conventions]

### Architecture constraints this story must satisfy

- **AD-4 — local-first via one durable outbox unit.** The client writes a watch commit to a local `expo-sqlite` table (`pending_watches`) **synchronously before any network call**; that same row carries nullable rating/mood/note columns (so Epic 3 doesn't need a local schema migration to add them later). A sync worker drains the outbox to `watches` via PostgREST — if the row is still pending, it's a single insert (here: idempotent upsert) carrying commit + rating/mood/note together. This is **not a general offline-first sync framework** — NFR8 caps v1 offline at "basic, not optimized." [Source: ARCHITECTURE-SPINE.md#AD-4]
- **AD-3 — watch is the atomic timestamped unit.** Rating, mood, note are columns on `watches`, never on a `titles`/catalog entity — there is no local titles table (AD-3, AD-6 note). Each logged watch is its own row. [Source: ARCHITECTURE-SPINE.md#AD-3]
- **AD-8 — every FK to `auth.users.id` is `ON DELETE CASCADE`.** `watches.user_id` cascades so a future `delete-my-account` (Epic 7) unwinds it structurally, not via a hand-maintained loop. [Source: ARCHITECTURE-SPINE.md#AD-8]
- **AD-1 — deny-by-default RLS, owner-scoped for now.** `watches` is explicitly named in AD-1's Binds list. RLS `select`/`insert`/`update`/`delete` all `user_id = auth.uid()` in 1.5 — the follower-visibility OR-clause is Story 1.6's job, not this story's. [Source: ARCHITECTURE-SPINE.md#AD-1]
- **ARCH-10 consistency conventions:** `tmdb_id` + `media_type` (`'movie'|'tv'`) — never a synonym column; `watches` is the one table that additionally carries `tmdb_episode_id` (episode-level granularity) per the spine's own callout. DB `snake_case`, TS `camelCase`. Ids: `uuid` via `gen_random_uuid()` on the server row (but see the "client-generated id" trap below). Timestamps: `timestamptz`, UTC, ISO 8601 at the API boundary. Rating: `smallint` half-steps 0–10. Moods: `text[]` — CHECK-constrained **once populated** (Epic 3), not now. [Source: ARCHITECTURE-SPINE.md#Consistency Conventions]

### Key design decisions / traps

- **Client-generated `id`, not server-generated — this is the idempotency mechanism (AC7).** `watches.id uuid primary key default gen_random_uuid()` is a *default for a direct-insert path*, but the sync worker must **always supply the id explicitly** (the same uuid the local `pending_watches` row was created with) and use `upsert(..., { onConflict: 'id' })`. If you let the server generate the id on insert instead, a retry after a lost response duplicates the row — exactly the failure Story 1.4 hit with `catalog_cache` batch upserts and fixed by de-duping. Generate the uuid client-side at `logWatch` time (`crypto.randomUUID()` / `expo-crypto`), never server-side, for any row that goes through the outbox.
- **`expo-sqlite`'s API surface changed materially across SDK generations — use the current async API, not the deprecated WebSQL-style `transaction()` callback.** Confirmed for SDK 56: `SQLite.openDatabaseAsync`/`openDatabaseSync`, `db.runAsync`/`execAsync` (writes), `db.getAllAsync`/`getFirstAsync`/`getEachAsync` (reads), `db.withTransactionAsync`/`withExclusiveTransactionAsync`. There is also `expo-sqlite/kv-store` (an AsyncStorage-shaped drop-in) — **do not use it here**; `pending_watches` needs real relational queries (`WHERE synced_at is null`), not a key-value store. [Verified against docs.expo.dev/versions/v56.0.0/sdk/sqlite/ at story-creation time — re-verify at implementation time in case of a point-release change.]
- **Sync is fire-and-forget from the UI's perspective — `logWatch` must never await the network.** The commit-is-instant guarantee (FR14, NFR1's "network irrelevant to the commit") is only real if the local insert alone gates the confirmation. Structure `logWatch` so its returned promise resolves right after `db.runAsync(insert...)`, and call `triggerSync()` without awaiting it (`void triggerSync()` or `.catch(() => {})` on the floating promise) so a slow/failed sync attempt can never delay or fail the UI's confirmation.
- **`watched_at` is always "now" in 1.5 — no backdating UI.** The story is the walking-skeleton log action off a search result, not the richer log flows (tracked-show ✓-Watched with pointer advance = FR11/Epic 3; bulk season log = FR13/Epic 3; both explicitly out of Epic 1). Don't build a date picker.
- **Mood/rating/note columns exist and stay null — resist the urge to stub a rating UI "since the columns are right there."** Epic 3 (FR17-21) owns that surface; wiring even a minimal rating tap here is scope creep this story doesn't need and Epic 3's story will assume doesn't exist yet.
- **`@react-native-community/netinfo` is a new native-module dependency (not an Expo-managed one) — it's the standard, documented way to get connectivity-change events in an Expo app (Expo's own docs reference it), unlike `expo-network` which as of this writing only offers a point-in-time state fetch, not a change listener.** If at implementation time Expo SDK 56 ships an equivalent listener API in a first-party module, prefer that instead — but do not skip the reconnect-trigger requirement (AC3) to avoid adding the dependency; a foreground-only trigger (Task 3b) alone leaves a phone that stays foregrounded through a reconnect never syncing.
- **Confirmation is a transient UI element, not a modal — do not block interaction.** FR14 says commit is "never blocked by rating or catalog latency"; by extension the confirmation itself must not block. A simple auto-dismissing banner (a few seconds) or an inline state swap near the search field is sufficient; there is no dedicated "toast" component in DESIGN.md to match against, so style consistent with the screen's existing `errorState` treatment.

- **Watch for duplicate-row / batch-abort failure modes under retry.** 1.4's real production defect (found in review) was exactly this class of bug: un-deduped upserts on `catalog_cache` threw "ON CONFLICT … cannot affect row a second time" and silently dropped the whole batch. `watchSync.ts`'s per-row upsert loop (Task 3) sidesteps the *batch* version of this (one row at a time, one failure doesn't abort the others) — but still get the `onConflict` key exactly right, since a wrong/missing conflict target reproduces the same class of bug. [Source: 1-4 Review Findings]
- **Timeouts matter — every network call in this codebase races a bounded timeout, not the platform default.** `catalog.ts`'s `searchCatalog` races a 10s timeout; `supabaseClient.ts`'s health check uses `AbortController` (RN's `AbortSignal.timeout()` is unavailable — build it from `AbortController`+`setTimeout` instead, same trap noted there applies to any `watchSync.ts` fetch/upsert call if you add one). Don't let a hung sync attempt block a future trigger indefinitely — the `syncing` guard flag (Task 3) should be released in a `finally`. [Source: 1-4, app/data/supabaseClient.ts]
- **Git context:** recent commits are 1.4 (`44f6aca feat: 1.4`) and its TMDB-key follow-up (`ceee5a7`) — no local-storage/outbox code exists yet anywhere in the client. This story writes the first of it. [Source: git log]
